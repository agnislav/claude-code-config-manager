## Phase 1 — QuickPick Multi-Select Filter

**Requirements:** FILT-01, FILT-02, FILT-03, FILT-04, FILT-05, FILT-06, FILT-07, FILT-08, FILT-09, FILT-10

**Goal:** Replace the 8 toolbar icon buttons (All + 7 sections) with a single filter icon that opens
a `vscode.createQuickPick` multi-select picker.

**Plans:** 2 plans

Plans:
- [x] 01-01-PLAN.md — Remove old filter infrastructure (commands, context keys, toolbar entries)
- [x] 01-02-PLAN.md — Implement QuickPick filter command with icon swap and TreeView description

### Success Criteria

1. The toolbar shows exactly one filter icon button; clicking it opens a multi-select QuickPick.
2. Opening the QuickPick when filters are active pre-selects the active sections; Escape leaves tree unchanged.
3. Selecting a subset filters the tree; accepting empty selection shows all sections.
4. After VS Code restarts, the filter resets to "All".
5. All `claudeConfig_filter_*` context keys are absent; no orphaned icon-toggle buttons remain.

## Phase 3 — User Scope Lock Toggle

**Requirements:** LOCK-01, LOCK-02, LOCK-03, LOCK-04, LOCK-05, LOCK-06, LOCK-07, LOCK-08, LOCK-09, LOCK-10

**Goal:** Add a lock toggle making User scope read-only for the session, with icon swap, write guards,
and lock-aware move/copy pickers.

**Plans:** 2 plans

Plans:
- [x] 03-01-PLAN.md — Lock state infrastructure: ConfigStore API, effectiveReadOnly propagation, ScopeNode contextValue
- [x] 03-02-PLAN.md — Commands and guards: lock/unlock commands, inline buttons, write command guards

### Success Criteria

1. A lock icon in the toolbar toggles between `$(lock)` and `$(unlock)` on click.
2. While locked, write operations to User scope display an informative message and make no change.
3. While locked, User scope does not appear as a selectable target in move/copy pickers.
4. File-watcher reload does not change lock state.
5. After VS Code restarts, lock resets to unlocked.

## Phase Details

### Phase 6: Lock UX Rework
**Goal**: Users experience lock-by-default protection with discoverable toolbar controls and correct icon semantics
**Depends on**: Phase 5 (lock infrastructure exists)
**Requirements**: LOCK-11, LOCK-12, LOCK-13
**Success Criteria** (what must be TRUE):
  1. On extension activation, the User scope is locked immediately — write operations are blocked before the user touches anything
  2. When the User scope is locked, the toolbar shows `$(lock)` (state: locked); when unlocked, it shows `$(unlock)` (state: unlocked)
  3. The lock toggle button appears in the toolbar (view/title), positioned left of the filter button — not on the User scope node inline
  4. Clicking the toolbar lock button still toggles the lock state and updates the icon atomically with no reload required
**Plans**: 1 plan

Plans:
- [ ] 06-01-PLAN.md — Lock-by-default activation, icon semantics, and toolbar migration (LOCK-11, LOCK-12, LOCK-13)

### Phase 7: Collapse/Expand Toolbar Buttons
**Goal**: Users can collapse or fully expand the entire tree with one click
**Depends on**: Phase 6
**Requirements**: TOOL-01, TOOL-02
**Success Criteria** (what must be TRUE):
  1. A Collapse All button appears in the toolbar; clicking it collapses all expanded nodes so only top-level scope nodes are visible
  2. An Expand All button appears in the toolbar; clicking it expands all nodes to full depth so every setting, rule, and entry is visible
  3. Both buttons work correctly regardless of current filter state or lock state
  4. The toolbar button order is consistent and predictable (lock, collapse, expand, filter)
**Plans**: 1 plan

Plans:
- [ ] 07-01-PLAN.md — Register Collapse All and Expand All toolbar commands with tree traversal expand logic (TOOL-01, TOOL-02)

### Phase 8: Object Settings Expansion
**Goal**: Users can inspect object-type settings by expanding them to see individual key/value pairs
**Depends on**: Phase 7
**Requirements**: SETT-01, SETT-02
**Success Criteria** (what must be TRUE):
  1. Object-type settings (those whose value is a non-scalar object with nested keys) render as expandable tree nodes — they have a collapse/expand arrow instead of showing `{N keys}` as a dead-end description
  2. Expanding an object setting reveals child nodes where each child displays the key as its label and the value as its description
  3. Scalar settings (strings, numbers, booleans) are unaffected — they continue to render as non-expandable leaf nodes
  4. Object settings with zero keys show as an expandable node that expands to nothing (empty), not as a `{0 keys}` label
**Plans**: 1 plan

Plans:
- [ ] 08-01-PLAN.md — Make object settings expandable with key/value child nodes (SETT-01, SETT-02)
