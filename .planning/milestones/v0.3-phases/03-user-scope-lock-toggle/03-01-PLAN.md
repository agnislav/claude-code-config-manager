**In `src/tree/nodes/scopeNode.ts`:**

1. Override `computeContextValue()` to include scope identity in the contextValue. Currently it appends `.missing` to `super.computeContextValue()`. Change to build the full contextValue with scope identity:
   ```typescript
   protected computeContextValue(): string {
     const base = `scope.${this.scopedConfig.scope}`;
     const editability = this.scopedConfig.isReadOnly ? 'readOnly' : 'editable';
     const parts = [base, editability];
     if (!this.scopedConfig.fileExists) parts.push('missing');
     return parts.join('.');
   }
   ```
   This produces values like: `scope.user.editable`, `scope.user.readOnly`, `scope.projectShared.editable`, `scope.projectShared.editable.missing`, etc.

   **Why this matters (LOCK-08):** The inline lock/unlock buttons need `when` clauses that match ONLY the User scope node. The current pattern `scope.editable` does not distinguish User from other scopes. With `scope.user.editable`, the regex `viewItem =~ /^scope\.user/` targets only User scope.

   **IMPORTANT:** This changes the contextValue format for ALL ScopeNode instances. The existing `when` clauses in `package.json` that match `viewItem =~ /^scope\\.*/` still work because `scope.user.editable` starts with `scope.` — but the `openFile` command's when clause `viewItem =~ /^scope\\./` still matches. The `createConfigFile` when clause `viewItem =~ /^scope\\..+\\.missing/` still matches `scope.user.editable.missing`. Verify all existing when clauses still match the new format.

2. Import `LOCK_URI_SCHEME` from `'../lockDecorations'` (the new file created below).

3. In the `ScopeNode` constructor, after setting `this.tooltip`, add a `resourceUri` for User scope nodes to enable lock decoration dimming:
   ```typescript
   // Set resourceUri for lock dimming decoration (User scope only)
   if (scopedConfig.scope === ConfigScope.User) {
     this.resourceUri = vscode.Uri.from({
       scheme: LOCK_URI_SCHEME,
       path: '/user',
       query: scopedConfig.isReadOnly ? 'locked' : 'unlocked',
     });
   }
   ```
   This must be placed BEFORE `this.finalize()`. The `scopedConfig.isReadOnly` here is the effectiveReadOnly value (already shallow-cloned in ConfigTreeProvider if locked). When locked, `isReadOnly` is `true` → query is `'locked'` → FileDecorationProvider dims the label.

   Note: Setting `resourceUri` on a TreeItem can affect the displayed label if the URI has a meaningful `fsPath`. Using a custom scheme (`claude-config-lock`) ensures VS Code does not try to resolve it as a file path. The label remains the explicit `SCOPE_LABELS[scope]` set via the `super()` call.

**Create `src/tree/lockDecorations.ts` (new file):**

Model this on the existing `PluginDecorationProvider` in `pluginNode.ts`:

```typescript
import * as vscode from 'vscode';

export const LOCK_URI_SCHEME = 'claude-config-lock';

export class LockDecorationProvider implements vscode.FileDecorationProvider {
  // Static undefined — no dynamic change notifications needed.
  // Lock toggle triggers tree refresh → ScopeNode rebuilt with new resourceUri.query
  // → VS Code calls provideFileDecoration for the new URI automatically.
  readonly onDidChangeFileDecorations = undefined;

  provideFileDecoration(uri: vscode.Uri): vscode.FileDecoration | undefined {
    if (uri.scheme !== LOCK_URI_SCHEME) return undefined;

    if (uri.query === 'locked') {
      return {
        color: new vscode.ThemeColor('disabledForeground'),
      };
    }
    return undefined;
  }
}
```

Use `'disabledForeground'` ThemeColor — this is already used by `PluginDecorationProvider` for disabled plugins and matches the codebase convention for inactive/dimmed state. Per user's discretion note: "should match VS Code's existing 'ignored' or 'disabled' color conventions."

The `onDidChangeFileDecorations = undefined` pattern matches `PluginDecorationProvider` exactly. No EventEmitter needed because lock toggle fires `configStore.onDidChange` → `treeProvider.refresh()` → tree fully rebuilds → `ScopeNode` reconstructed with updated `resourceUri.query` → VS Code calls `provideFileDecoration()` for the new URI.
