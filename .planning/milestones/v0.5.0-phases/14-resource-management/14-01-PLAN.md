  <name>Task 1: Implement Disposable on ConfigTreeProvider and register in subscriptions (RES-01)</name>
  <files>src/tree/configTreeProvider.ts, src/extension.ts</files>
  <action>
In `src/tree/configTreeProvider.ts`:
1. Add `vscode.Disposable` to the `implements` clause on the ConfigTreeProvider class declaration:
   `export class ConfigTreeProvider implements vscode.TreeDataProvider<ConfigTreeNode>, vscode.Disposable`
2. Add a `dispose()` method to ConfigTreeProvider that calls `this._onDidChangeTreeData.dispose()`:
   ```
   dispose(): void {
     this._onDidChangeTreeData.dispose();
   }
   ```
   Place it after the `refresh()` method for logical grouping.

In `src/extension.ts`:
3. Add `treeProvider` to the `context.subscriptions.push(...)` call. It should be added alongside the existing disposables (after `treeView` is a natural position, since it was created right after treeProvider). The existing line:
   ```
   context.subscriptions.push(
     treeView, configStore, fileWatcher, ...
   ```
   becomes:
   ```
   context.subscriptions.push(
     treeView, treeProvider, configStore, fileWatcher, ...
   ```

This ensures VS Code calls `treeProvider.dispose()` during extension deactivation, which disposes the `_onDidChangeTreeData` EventEmitter and prevents memory leaks across Extension Host restart cycles.
  </action>
  <verify>
Run `npm run typecheck` to confirm the Disposable interface is satisfied and no type errors introduced. Grep for `dispose()` in configTreeProvider.ts to confirm the method exists. Grep for `treeProvider` in the subscriptions push call in extension.ts.
  </verify>
  <done>ConfigTreeProvider implements vscode.Disposable with a dispose() method that disposes the EventEmitter. The treeProvider instance is registered in context.subscriptions so VS Code disposes it on deactivation.</done>

  <files>src/config/configModel.ts</files>
  <action>
In `src/config/configModel.ts`:
1. Add import for PluginMetadataService at the top:
   `import { PluginMetadataService } from '../utils/pluginMetadata';`

2. In the `reload()` method, call `PluginMetadataService.getInstance().invalidate()` at the start of the method body (before any config loading occurs). This ensures both full reloads and single-folder reloads get fresh plugin metadata.

3. Add a `console.log` for cache invalidation debugging, matching the existing log format used in Phase 12/13:
   `console.log('[Claude Config] Plugin metadata cache invalidated on reload');`
   Place the log line immediately after the invalidate() call.

The `invalidate()` method already exists on PluginMetadataService (clears both `registry` and `manifestCache`). Calling it in reload() ensures:
- Fresh plugin data after config changes (enable/disable sequences)
- Fresh data after external config file changes (file watcher triggers reload)
- No stale descriptions or install paths displayed in the tree

Do NOT modify PluginMetadataService itself — the `invalidate()` method is already correctly implemented.
  </action>

Run `npm run typecheck` to confirm the import resolves and no type errors. Grep for `PluginMetadataService` in configModel.ts to confirm the import and invalidate call exist. Run `npm run compile` to verify the full build succeeds.
  </verify>
  <done>ConfigStore.reload() invalidates the plugin metadata cache before reloading config files. Cache invalidation is logged to console for debugging. Plugin descriptions and install paths reflect the latest state after every reload, whether triggered by user action or file watcher.</done>
