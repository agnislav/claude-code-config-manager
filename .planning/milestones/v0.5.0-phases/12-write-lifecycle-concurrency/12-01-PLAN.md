<task type="auto">
  <name>Task 1: Add in-flight write tracking and output channel logging to configWriter</name>
  <files>src/config/configWriter.ts, src/constants.ts</files>
  <action>
Add timing constants to `src/constants.ts`:
- `DEBOUNCE_RELOAD_MS = 300` — regular file watcher debounce delay
- `DEBOUNCE_MAX_WAIT_MS = 2000` — maximum wait ceiling for debounce
- Add a doc comment for each constant explaining its purpose

In `src/config/configWriter.ts`, add write lifecycle tracking infrastructure:

1. Create a module-level `Set<string>` called `inFlightPaths` that tracks file paths currently being written.

2. Create a module-level `outputChannel` variable of type `vscode.OutputChannel | undefined` (not initialized — set via init function).

3. Export an `initWriteTracker(channel: vscode.OutputChannel)` function that stores the output channel reference.

4. Export a `isWriteInFlight(filePath: string): boolean` function that checks if a path is in the in-flight set.

5. Export a `getInFlightWriteCount(): number` function that returns `inFlightPaths.size` (used by deactivation logic to know when all writes are done).

6. Create a private `logWrite(message: string)` helper that writes to the output channel (if set) with a `HH:MM:SS.mmm` wall-clock timestamp prefix. Format: `[HH:MM:SS.mmm] [write] {message}`.

7. Create a private `trackedWrite(filePath: string, writeFn: () => void): void` wrapper function that:
   - Adds `filePath` to `inFlightPaths`
   - Logs `"start: {filePath}"` via logWrite
   - Records `Date.now()` for duration tracking
   - Calls `writeFn()` in a try block
   - On success: logs `"complete: {filePath} ({duration}ms)"`
   - On error: logs `"FAIL: {filePath} ({duration}ms) — {error.message}"`, then re-throws
   - In `finally` block: always removes `filePath` from `inFlightPaths`, logs `"watcher resumed: {filePath}"`

8. Wrap every `writeJsonFile()` call in configWriter.ts with `trackedWrite()`. There are write calls in: `addPermissionRule`, `removePermissionRule`, `setEnvVar`, `removeEnvVar`, `setScalarSetting`, `removeScalarSetting`, `addHookEntry`, `removeHookEntry`, `setMcpServer`, `removeMcpServer`, `setPluginEnabled`, `removePlugin`, `setSandboxProperty`. Each call should become:
   ```typescript
   trackedWrite(filePath, () => writeJsonFile(filePath, config));
   ```

Do NOT add retry logic — fail fast per user decision. The existing error handling in command callers (showWriteError) is sufficient.

<task type="auto">
  <name>Task 2: Add watcher suppression and maxWait debounce ceiling to fileWatcher</name>
  <files>src/watchers/fileWatcher.ts, src/extension.ts</files>
  <action>
Update `src/watchers/fileWatcher.ts` to support in-flight write suppression and maxWait:

1. Import `isWriteInFlight` from `../config/configWriter` and `DEBOUNCE_RELOAD_MS`, `DEBOUNCE_MAX_WAIT_MS` from `../constants`.

2. Add a private `maxWaitTimeout: ReturnType<typeof setTimeout> | undefined` field alongside the existing `reloadTimeout`.

3. Add a private `outputChannel: vscode.OutputChannel | undefined` field.

4. Add a `setOutputChannel(channel: vscode.OutputChannel): void` method to store the output channel reference.

5. Replace the `debouncedReload()` method with a new implementation:
   - Accept an optional `filePath?: string` parameter (the changed file path, if available from watcher callbacks).
   - If `filePath` is provided and `isWriteInFlight(filePath)` returns true, log `"[HH:MM:SS.mmm] [watcher] suppressed reload: {filePath} (write in-flight)"` to the output channel and return immediately (skip reload).
   - Otherwise, implement debounce + maxWait:
     - Clear existing `reloadTimeout` if set.
     - Set `reloadTimeout` to `setTimeout(doReload, DEBOUNCE_RELOAD_MS)` where `doReload` calls `this.configStore.reload()`, clears `reloadTimeout`, and clears `maxWaitTimeout`.
     - If `maxWaitTimeout` is NOT already set, set it to `setTimeout(doReload, DEBOUNCE_MAX_WAIT_MS)`. This ensures that during rapid changes, a reload happens at most every 2 seconds even if changes keep extending the regular debounce.

6. Update all watcher callback lambdas in `watchPattern()` and `watchAbsolute()` to pass the changed file URI's fsPath to `debouncedReload()` where available. For `watchPattern()`, the `onDidChange`, `onDidCreate`, and `onDidDelete` events provide a `vscode.Uri` argument — pass `uri.fsPath` to `debouncedReload(uri.fsPath)`. For `watchAbsolute()`, same pattern. For the workspace folder change listener, call `debouncedReload()` with no argument (no file path available).

7. Update `dispose()` to also clear `maxWaitTimeout` if set.

Wire it up in `src/extension.ts`:
- After creating `outputChannel` (line 21) and before `fileWatcher.setup()` (line 119):
  - Call `initWriteTracker(outputChannel)` (import from `./config/configWriter`)
  - Call `fileWatcher.setOutputChannel(outputChannel)` to pass the output channel to the watcher

Do NOT modify the deactivation logic in this task — that is handled in Plan 02.
  </action>

Verify `maxWaitTimeout` is created when debounce starts and cleared on reload or dispose. Verify watcher callbacks pass file paths. Verify extension.ts wires up `initWriteTracker` and `setOutputChannel`.</verify>
  <done>File watcher suppresses reloads for in-flight write paths. MaxWait ceiling ensures reload within 2000ms during continuous changes. Output channel receives suppression logs. Extension.ts wires up the output channel to both configWriter and fileWatcher.</done>
